# Решение проблемы "Invalid page in block XXX of relation YYY".

Пример: `invalid page in block 460248 of relation base/16391/23665`.
Это всегда означает какую-то аппаратную проблему на системе хранения, где СУБД хранит свои данные.

## Cохранить то, что имеется, если резервной копии у клиента по каким-то причинам нет

1) Отключить пользователей от базы данных, чтобы предотвратить её дальнейшее разрушение.
2) Посмотреть какая сущность у нас развалилась: `SELECT 23665::regclass::text;`
3) Если это индекс, просто удаляем его и пытаемся снять полный дамп базы данных, записав какой индекс мы удалили.
4) Если это таблица, то определить строки с какими ID теперь недоступны и сдампить таблицу пока без них:
   ```sql
   SET enable_seqscan = OFF;
   COPY (
     SELECT * FROM some_items WHERE id < 100500 UNION ALL
     SELECT * FROM some_items WHERE id > 100500 AND id < 100800 UNION ALL
     SELECT * FROM some_items WHERE id > 100800
   ) TO '/tmp/some_items-table.dump'
   ```
   На репликах данные могут развалиться по-разному. Из них можно попробовать наскрести недостающие куски. Их можно снять через тот же COPY с запросом SELECT
   Если реплики нет, можно посмотреть какие индексы есть на сбойной таблице. Не исключено, что критичные столбцы покрыты индексами, которые остались целыми.
   Данные из индексов можно вытащить, сделав запрос так, чтобы он затрагивал только столбцы, которые есть в индексе. Если в интересующем нас индексе нет ключевого столбца, не беда, там есть технический столбец ctid, по которому потом можно соединить данные построчно.
   Если строки восстановить не удалось, а на таблицу ссылаются другие таблицы, надо заполнить недостающее любым мусором, чтобы внешним ключам было куда ссылаться.
6) После того как мы определили и сдампили все сбойные таблицы, можно сдампить остальную базу данных следующей командой:
   pg_dump --exclude-table-data=some_items -j10 -Fd -Z1 -f /dump_directory dbname

## Восстановить данные из дампов на другом серевере СУБД

1) Для начала восстанавливаем основную базу данных, но без индексов и внешних ключей:
   pg_restore -d dbname -Fd /dump_directory --section=pre-data --section=data -j10
2) Потом заливаем дампы отдельных таблиц, после чего доливаем индексы и внешние ключи:
   pg_restore -d dbname -Fd /dump_directory --section=post-data -j10
   Если всё сделано правильно, должна получиться консистентная база данных. 
3) Создать удалённые ранее индексы, если таковые были 
4) Если вы создавали мусорные строки, то надо понять как с ними быть дальше. Например, удалить их или заполнить значениями «данные утеряны».
